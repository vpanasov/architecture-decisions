# RFC-0009: Unified Mobile POS — объединение retail-app и fortis-pos-app на базе единой модульной платформы

> **Источник:** [Confluence](https://bz.incdb.team/pages/viewpage.action?pageId=3419832833)

**Статус:** DRAFT.
**Автор:** Арсен Тер-Оганисян.
**Дата:** 16 апр. 2026 г.

Проблема
--------

### Контекст

Компания эксплуатирует **два отдельных Android POS-приложения** с непересекающимися кодовыми базами:

* **retail-app** (`ru.lifepay`) — Россия: ФНС/ОФД, ЕГАИС, маркировка, СБП, BNPL, PIN-аутентификация, 8+ ERP-интеграций
* **fortis-pos-app** (`world.fortis`) — MENA (ОАЭ): AANI, ресторанные заказы, лояльность, чаевые, аутентификация по номеру телефона

| Метрика | Значение |
| --- | --- |
| Общий код | 27 фич + 11 флоу идентичны в обоих приложениях — поддерживаются независимо, без reuse |
| Reuse Rate | 0% (не считая общей дизайн-системы) — каждая фича разрабатывается и фиксируется дважды |
| Баги | Дублируются — один и тот же баг фиксируется в двух кодовых базах |
| Совместимость стеков | Техническая ~80% (Kotlin, Jetpack Compose, Decompose, Retrofit, Room). Расхождения: навигация (Cicerone+Decompose vs Decompose), DI processor (kapt vs ksp), периферия (внешняя lib vs внутренняя sdk), конфигурация (статическая vs динамическая с KSP), аутентификация (PIN/MD5 vs phone+OTT) |
| Общий предок | Коммит `8c677b9a` от 2023-06-01. Расхождение — июль 2023, ~6 недель спустя |
| Кодовая база | 7 076 .kt файлов (retail-app: 4 027, fortis-pos-app: 3 049) |
| Новый рынок | Существенно затруднён |

### Влияние

**На кого влияет:**

* **IT** — двойные затраты на разработку и поддержку 27 общих фич и 11 флоу. Баги исправляются в двух местах независимо
* **Product** — два изолированных backlog'а и продуктовых процесса. Нет единого видения, нет переноса UX-решений и лучших практик между рынками
* **Бизнес** — масштабирование на новые рынки заблокировано

**Блокируемые инициативы:**

* Консолидация мобильного R&D для повышения эффективности
* Выход на новые рынки (KZ, UZ, SA и др.)
* Единый Design System и UI Kit для всех рынков
* Единая AI/Data платформа
* Public Mobile SDK для партнёров и интеграций

Варианты решения
----------------

### Вариант 1 -- Объединить два приложения в одно

Слить retail-app и fortis-pos-app в единую кодовую базу с гибкой конфигурацией рынка.

| Подвариант | Преимущество | Недостатки |
| --- | --- | --- |
| 1А -- За основу — Fortis POS | На первый взгляд быстрее — Fortis архитектурно чище (Decompose, KSP) | Продуктовые задачи блокируют платформенную разработку и наоборот. Монолитность и отсутствие мультиплатформенности не решены. |
| 1Б -- За основу — Life POS | RU compliance уже внутри | Те же, что у 1А. Дополнительно: kapt несовместим с Kotlin 2.x; гибридная Cicerone+Decompose усложняет development |

### Вариант 2 -- Новые приложения на базе общей платформы *(выбран)*

Создать Common SDK как отдельный слой. Рыночные приложения — тонкие оболочки поверх него.

| Что решает | Как |
| --- | --- |
| Дублирование | Один Common SDK для всех рынков — 27 фич и 11 флоу разрабатываются и фиксируются один раз |
| Масштабирование O(N) | Новый рынок = Market Plugin поверх SDK. TTM North Star: ~12 недель на новый рынок |
| Монолитность | Каждый модуль SDK опционален, поставляется как отдельная Gradle-зависимость |
| Мультиплатформенность | Бизнес-логика в platform-agnostic слое, UI-адаптеры под Android / Aurora / Web |
| Независимость команд | Platform и продуктовые команды развиваются без взаимных блокировок |

Решение
-------

Предлагается создать единую модульную мобильную платформу **Unified Mobile POS SDK**. Не мержим два приложения и не создаём новое с нуля — берём архитектурные паттерны fortis-pos-app как основу (Decompose, KSP, динамическая конфигурация) и переносим retail-app-специфику как подключаемые **Market Plugins**.

Мобильная архитектура строится на трёх уровнях:

* **Unified Platform SDK** — единая платформа, которая поставляется как набор Gradle-зависимостей и не знает ни о рынках, ни об индустриях. Состоит из двух подслоёв:
  + **Kernel** (`platform-core`) — техническая инфраструктура: DI (Dagger 2 + KSP), навигация (Decompose), сеть (Retrofit + OkHttp), база данных (Room), синхронизация (WorkManager), безопасность (Keystore), UI Kit (DesignTokens, BrandingConfig).
  + **Domain modules** (`common-sdk`) — 27 feature-модулей и 11 flow-модулей. Рыночно и индустриально нейтральны.
* **Market Plugin** (`markets/xx/core` + `markets/xx/ux`) — реализует Plugin Interfaces и содержит **всю специфику конкретного рынка и индустрии**.
* **Apps** (`apps/xx/`) — условные конфигурации и точки сборки. Не содержат бизнес-логики.

Ключевой принцип — **Modules, not Markets**: нигде в Domain modules нет понятия «рынок» или «индустрия». Граница проходит по Plugin Interfaces.

4 stream-aligned команды работают параллельно: **Platform** (platform-core + platform-libs + ui-kit), **Common SDK** (миграция 27 фич + 11 флоу), **RU Market** (FiscalPlugin, маркировка, ЕГАИС, российские платёжные драйверы), **AE Market** (AANI, ресторанная специфика, ближневосточные драйверы).

### Архитектура backend

Backend следует той же трёхуровневой модели, что и мобильное приложение.

| Уровень | Состав | Принцип |
| --- | --- | --- |
| Инфраструктурные сервисы *(Kernel backend)* | Авторизация и аутентификация, синхронизация состояния, безопасность и аудит, API Gateway и маршрутизация, Observability | Общая инфраструктура, не связанная с предметной областью |
| Доменные микросервисы *(Domain backend)* | По одному микросервису на домен: каталог, заказы, платежи, клиенты, инвентаризация, конфигурация и др. | Явные входящие зависимости, явный исходящий API |
| Рыночные адаптеры *(Market adapter)* | RU: ФНС/ОФД, CRPT (маркировка), российские платёжные системы, ERP-интеграции. AE: AANI, налоговые схемы ОАЭ | Вся регуляторная и индустриальная специфика — только здесь |

### Domain as product

**Допустимые бизнес-модели:** транзакционная (pay-per-use / API-calls); лицензирование SDK (OEM, SaaS, Enterprise); партнёрская (revenue share или white-label); консалтинг при внедрении; кастомная доработка SDK.

Образ результата
----------------

### 1. Платформа с независимым релизным циклом

Unified Platform SDK живёт как самостоятельный продукт с релизным циклом, независимым от регионов. Платформа фиксирует матрицу совместимости версий.

### 2. Независимые релизные циклы приложений

Конечные приложения для каждого региона имеют **независимые друг от друга** релизные циклы.

### 3. Типовой сценарий: приложение для нового рынка

1. Создать `apps/xx/` — Application layer, Gradle-конфигурация сборки.
2. Подключить актуальную версию Unified Platform SDK.
3. Написать `markets/xx/core/` — реализации Plugin Interfaces.
4. При необходимости добавить `markets/xx/ux/` — рыночные фичи и флоу.

Код Unified Platform SDK не трогаем. TTM North Star: ~12 недель на новый рынок.

Компромиссы
-----------

### 1. Тяжёлая миграция существующих приложений

Переход на новую платформу затрагивает оба существующих приложения — и fortis-pos-app, и retail-app. Это не технический рефакторинг, а полноценный миграционный проект.

### 2. Техдолг по схеме «market-first -> платформа»

Рыночная команда может реализовать функциональность в Market Plugin раньше, чем она появится в платформе. Позднее потребуется перенос в доменные модули SDK.

### 3. Необходимость функционального фриза обоих приложений

В период активной разработки платформы новые фичи в legacy-приложениях не добавляются — только критические исправления.

Ключевые тезисы, требующие согласования
---------------------------------------

1. **Это новый проект, а не адаптация существующих приложений**
2. **Платформа и рыночные команды — разделённые организационные единицы**
3. **Переход к микросервисной архитектуре с доменной ответственностью**
4. **Cross-platform с самого старта** — Android, Aurora OS, Web
5. **Разработка новых решений и миграция — последовательные, а не параллельные этапы**
