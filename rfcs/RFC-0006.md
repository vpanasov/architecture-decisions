1. Проблема
-----------

Текущий процесс code review полностью ручной. Это создаёт следующие проблемы:

* **Неоднородное качество ревью.** Глубина проверки зависит от загрузки и экспертизы конкретного ревьюера. Один и тот же тип ошибки может быть пойман в одном MR и пропущен в другом. Кросс-файловые проблемы — нарушения контрактов между модулями, дублирование логики — часто проходят незамеченными, потому что ревьюер видит только diff, а не полный контекст кодовой базы.
* **Нарушения конвенций.** Стилистические и архитектурные конвенции проверяются вручную. Нет единого автоматического baseline — результат зависит от того, кто ревьюит.
* **Дефицит времени сеньоров.** Качественное ревью требует вовлечения старших разработчиков. Их время — дефицитный ресурс, который расходуется на рутинные замечания (стиль, конвенции, очевидные ошибки) вместо фокуса на архитектурных и бизнес-логических проблемах.

**Цель:** проверить гипотезу, что AI-инструмент для code review способен обеспечить стабильный baseline качества проверки и разгрузить сеньоров от рутинной части ревью.

### Контекст

Компания проходит AI-трансформацию инженерных процессов. Команда LifePOS выбрана как пилотная — мы пробуем различные AI-инструменты в разных частях процесса разработки (code review, тестирование, разработка). AI code review — одно из направлений. Стоимость эксперимента минимальна: бесплатный триал, интеграция через webhook за 30 минут, отключение за минуты.

---

2. Предлагаемое решение
-----------------------

Подключить **Greptile** (AI code review инструмент) к корпоративному GitLab на бесплатный пилотный период **14 дней** для команды LifePOS.

### Что такое Greptile

Greptile — AI-агент для code review (Y Combinator, SOC 2 Type II). Ключевое отличие от аналогов: индексирует **всю кодовую базу** репозитория, строит граф зависимостей между функциями, классами и модулями. Это позволяет анализировать MR не изолированно по diff, а с учётом полного контекста проекта.

**Как работает индексация:** Greptile получает доступ к репозиторию через GitLab API (group access token). Код проходит три этапа обработки: (1) парсинг в AST — извлечение функций, классов, переменных; (2) построение направленного графа зависимостей — вызовы, импорты, связи между модулями; (3) генерация векторных эмбеддингов для семантического поиска. Индекс обновляется инкрементально при каждом новом коммите/MR.

Основные возможности:

* Автоматические inline-комментарии к MR с конкретными замечаниями и предложениями исправлений.
* Обнаружение кросс-файловых проблем: нарушения контрактов между модулями, дублирование кода, нарушения конвенций.
* Генерация саммари MR.
* Настраиваемые правила проверки (coding standards, архитектурные конвенции).

**Отказоустойчивость:** Greptile подключается через webhook и **не является блокирующим элементом pipeline**. Если сервис недоступен — AI-комментарий просто не появляется, MR не блокируется, ручное ревью продолжается в штатном режиме.

### Параметры пилота

| Параметр | Значение |
| --- | --- |
| Версия | Greptile v4 (cloud) |
| Срок | 14 дней (бесплатный триал, без привязки карты) |
| Команда-участник | LifePOS (разработка + автоматизация тестирования) |
| Репозитории | Retail App, LifePosAutotests |
| Интеграция | GitLab Self-Managed -> Greptile Cloud (webhook) |
| Кто оценивает результаты | Ника Зимневич (лид тестирования), Александр Заровный (лид разработки) |

### Критерии успеха пилота

1. **Address rate >= 30%** — доля комментариев Greptile, по которым разработчик внёс исправления в код до мержа.
2. **Положительная оценка команды** — соотношение thumbs up / thumbs down на комментариях Greptile > 2:1.
3. **Разгрузка сеньоров** — Зимневич и Заровный оценивают полезность ревью по 5-балльной шкале. Целевой средний балл >= 3.5.
4. **Полнота AI-ревью** — количество комментариев, добавленных ручными ревьюерами сверх Greptile: в среднем менее 1 нового замечания на MR.
5. **Отсутствие** инцидентов ИБ за время пилота.

### Что происходит после пилота

| Результат пилота | Действие |
| --- | --- |
| Критерии успеха выполнены | Повторный выход на архкомитет с предложением о платной подписке (Cloud или Self-Hosted) |
| Критерии не выполнены | Отключение интеграции, публикация результатов |

3. Почему не альтернативы
-------------------------

| Альтернатива | Почему не выбрана |
| --- | --- |
| **Ничего не делать** | Code review остаётся полностью ручным. Качество зависит от загрузки ревьюера, кросс-файловые проблемы не ловятся. |
| **Доработать существующий самописный скрипт** | В команде уже есть AI-ревью скрипт (lifeposautotests, ~113 строк). Ограничения: не знает контекст проекта, пропускает MR > 30 файлов, нет inline, нет поддержки правил и конвенций. |
| **CodeRabbit** | Работает по diff, без полного индексирования кодовой базы. Слабее в обнаружении кросс-файловых проблем. |
| **Sourcery** | Фокус на рефакторинге, читаемости и security-сканировании, а не на поиске логических багов. Нет self-hosted варианта. |
| **GitLab Duo** | Требует GitLab Premium/Ultimate. |

4. Что теряем (компромиссы и техдолг)
-------------------------------------

### Риски пилота

| Риск | Severity | Митигация |
| --- | --- | --- |
| **Код уходит в облако Greptile** | Высокий | SOC 2 Type II. Opt-out от использования данных для обучения AI. При масштабировании доступен self-hosted вариант. |
| **Vendor lock-in** | Низкий | Пилот 14 дней, интеграция через webhook — отключается за минуты. |
| **Шум от ложных срабатываний** | Средний | Настройка правил во время пилота. Мониторинг acceptance rate. |
| **Стоимость после пилота** | Средний | Пилот бесплатный. Решение о платной подписке — отдельный вопрос после оценки результатов. |

### Техдолг

Пилот не создаёт техдолга. Интеграция через webhook, не требует изменений в CI/CD pipeline или коде проектов. При отключении — удаление webhook и токена.
