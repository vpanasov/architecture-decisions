1. Проблема и контекст
----------------------

### Проблема

Цикл code review — один из узких мест доставки фич в Fortis. Ручное ревью зависит от загрузки ревьюера: MR висит в ожидании первого отклика часы или сутки; часть итераций уходит на рутину (стиль, очевидные ошибки, пропущенные проверки), а не на содержательные архитектурные замечания; кросс-файловые и кросс-репозиторные проблемы пропускаются, потому что ревьюер видит diff, а не полный контекст системы.

Задача: **сократить время прохождения ревью и ускорить delivery при сохранении и повышении качества кода**. Пилот Greptile на LifePOS (RFC-0006) подтвердил, что AI-ревью закрывает это на уровне отдельной команды. Встал вопрос масштабирования на всю инженерную организацию — 9+ репозиториев, разнородный стек (.NET 9 / Angular 20 / Python-FastAPI / Kotlin), compliance-периметр с ПД и платёжными данными мерчантов UAE/Middle East.

Все внешние SaaS-инструменты AI code review (Greptile, CodeRabbit, Sourcery, GitLab Duo) объединяет одно ограничение: пайплайн ревью — чёрный ящик поставщика. Компании недоступно углубить проверку, добавить доменные этапы, обучать систему на реакциях команды, управлять бюджетом токенов или интегрировать внутренние источники контекста.

### Связь с RFC-0006

RFC-0006 — пилот внешнего SaaS на одной команде. Данный RFC — решение о долгосрочной архитектуре code review для всей организации.

2. Предлагаемое решение
-----------------------

Развернуть собственный сервис `fortis-mr-reviewer` в Yandex Cloud Managed Kubernetes как single-container FastAPI-приложение, подключённое к `gitlab.life-pay.ru` webhook'ом на уровне группы.

### 2.1. Пайплайн под нашим контролем

Каждый этап ревью реализован внутри сервиса. Базовый пайплайн:

```
GitLab webhook
  -> trigger.detect_trigger() -> ReviewTask | ClassifierTask
  -> Redis Streams queue (персистентность при рестарте pod)
  -> Worker:
    1. git clone/fetch + временный worktree
    2. pre-fetch контекста параллельно
    3. UnifiedReviewer agent (Claude CLI, structured output)
    4. Deep-dive subagents (до 5 параллельно)
    5. Publish в GitLab: inline-комментарии, replies, resolve, reactions, summary
    6. Record в Postgres: hashes, feedback, analytics
```

### 2.2. Агент работает в изолированном worktree с полным доступом к коду

На каждый MR `git_ops` клонирует/fetch'ит репозиторий в уникальный worktree внутри pod'а. Агент получает tool use (`Read`, `Grep`, `Glob`, безопасные `git` subcommand'ы) и ходит по коду как человек-ревьюер. Worktree удаляется после ревью.

### 2.3. Контроль действий агента через structured output

Все агенты возвращают JSON по предопределённой схеме (`runner_schemas.py`). Модель не публикует в GitLab напрямую — Python-слой принимает валидный JSON и выполняет публикацию с дополнительными проверками.

### 2.4. Обучение: project/tech memory + Consolidator + ветка `bot/memory`

Правила и накопленная экспертиза хранятся в репозитории самого ревьюера:

* `patterns/_defaults/` — базовый baseline
* `patterns/_security/policy.md` — политики ИБ
* `patterns/<repo-slug>/profile.json` — стек и конвенции конкретного проекта
* `patterns/<repo-slug>/memory.md` — живая память по проекту
* `patterns/_shared/` — кросс-проектные уроки

**Consolidator** — отдельный агент, запускается по триггеру (накопился объём feedback). Коммитит изменения в ветку `bot/memory`.

### 2.5. Интеграция с доменным контекстом (hypothesis-driven)

Гипотезы: MCP-сервер к Confluence (база знаний), API-контракты, связанные репозитории (read-only worktree). Все направления — тестируемые гипотезы.

### 2.6. Гибкость интеграций

MCP-серверы (Confluence, Kaiten, Sentry), bot с правами разработчика, доменные tool use, per-repo конфигурация через `.ai/settings.yml`.

### 2.7. Контроль токенов и расходов

Все LLM-вызовы логируются: per-MR, per-repo, per-role.

### 2.8. Отказоустойчивость и безопасность

* Webhook неблокирующий для GitLab pipeline
* Redis Streams сохраняют очередь при рестарте pod
* Kill switch: `worker_concurrency=0`
* Non-root pod, секреты через Vault, webhook защищён `X-Gitlab-Token`

3. Альтернативы
---------------

| Альтернатива | Почему не выбрана |
| --- | --- |
| Ничего не делать / остаться на Greptile | Пайплайн — чёрный ящик. Нет глубокой кастомизации. |
| Самописный скрипт `lifeposautotests` | ~113 строк, без контекста, пропускает MR > 30 файлов |
| CodeRabbit / Sourcery | Только по diff, ещё меньше контроля |
| GitLab Duo | Требует Premium/Ultimate, фиксированный движок |
| Greptile self-hosted | Платная подписка. Статический индекс без tool use |
| **`fortis-mr-reviewer` (выбрано)** | Полный контроль пайплайна, tool use в worktree, structured output, обучаемость |

4. Компромиссы и технический долг
---------------------------------

### Компромиссы

* **Зависимость от Anthropic API** — сознательный vendor-lock уровня LLM-провайдера.
* **Owner и дежурство** — в отличие от SaaS, сбой решаем мы.
* **В Anthropic уходят запрошенные агентом сниппеты** — объём виден в логах; DPA и SOC 2 Type II у провайдера есть.

### Технический долг

| Описание долга | Как планируем закрывать |
| --- | --- |
| Нет автоматических regression-тестов на поведение агентов | Golden-dataset из реальных MR |
| Интеграция с БЗ, API-контрактами, связанными репозиториями | Серия экспериментов по мере стабилизации |

5. Ресурсы и нагрузка
---------------------

* Текущий объём: ~30 MR/сутки по всем репозиториям Fortis на пике
* Pod: 1 CPU, 1 GiB RAM baseline, горизонтальный scale-out через Redis Streams

7. План реализации
------------------

* **Этап 1 (текущий -> 2026-05-01):** dev. Подключены `fortis-backend`, `fortis-backend-admin`, `fortis-web-app`.
* **Этап 2 (2026-05-01 -> 2026-05-15):** qa. Подключаем остальные репозитории. Параллельная работа с Greptile на LifePOS.
* **Этап 3 (2026-05-15 -> 2026-06-01):** preprod -> production. Решение по Greptile.
* **Этап 4 (дальше):** направление развития определяется данными.

### Критерии готовности

* Подключены все ревью-релевантные репозитории.
* Address rate >= 40%
* Thumbs up/down > 3:1
* Оценка лидов команд >= 4.0 / 5.0
* Ни одного ИБ-инцидента за первый квартал.

8. Метрики успеха
-----------------

* Среднее время `MR open -> первый комментарий бота/ревьюера`
* Среднее время `MR open -> merged`
* Доля MR, замерженных с поправками по AI-комментариям (address rate)
* Latency от webhook до первого комментария: p50 < 60 сек, p95 < 300 сек

9. Влияние на другие команды / юниты
------------------------------------

| Кто затронут | Как именно | Согласование |
| --- | --- | --- |
| Все команды разработки Fortis | Inline-комментарии от бота в MR. Онбординг ~15 мин. | В процессе |
| DevOps | Helm values (Yandex Cloud), GitLab group-webhook, Vault policy | В процессе |
| ИБ | Threat model, DPA Anthropic, аудит tool use | В процессе |
| Compliance | Проверка отсутствия утечки чувствительных данных в LLM-запросах | В процессе |
