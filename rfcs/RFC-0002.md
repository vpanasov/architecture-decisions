Проблема
--------

### Контекст

Компания эксплуатирует **два отдельных мерчант-портала** с непересекающимися кодовыми базами:

* **Fortis** (MENA) — задумывался как глобальный, но на данный момент захардкожен под один рынок (Дубай). Keycloak (OAuth2/OIDC), BFF pattern, cursor pagination, structured data model, Fee system, VAS (Loyalty, Menu, Shop, Appointments, AI Assistant)
* **LP** (Россия) — Custom JWT (phone+OTT), API v6 (PascalCase, page-based pagination, guid), фискализация, маркировка, ЕГАИС, 8+ ERP-интеграций, опросы

| Метрика | Значение |
| --- | --- |
| **FTE** | 206 (LP: 104+8 вакансий, Fortis: 51+43 вакансии) |
| **Reuse Rate** | 0% — каждая фича пишется дважды |
| **Баги** | Дублируются — один и тот же баг фиксится в двух кодовых базах |
| **Совместимость стеков** | Техническая ~80% (Angular 20, Standalone). Расхождения: аутентификация (Keycloak vs Custom JWT), VAS-набор (разные сервисы), compliance (фискализация, маркировка, ЕГАИС), ERP-интеграции (8+ в LP vs 0 в Fortis), локализация |
| **Новый рынок** | Невозможен — даже Fortis (задумывавшийся глобальным) захардкожен под один регион |

Технический аудит показывает ~80% совместимости стеков (оба на Angular 20, Standalone components), но продуктовая совместимость значительно ниже. Расхождения: аутентификация (Keycloak vs Custom JWT), API-конвенции (BFF vs versioned REST), pagination (cursor vs page-based), data model (structured vs flat), разные наборы VAS (Loyalty/Menu/Shop/AI в Fortis vs Surveys/CRM в LP), разная compliance-специфика (фискализация, маркировка, ЕГАИС — только LP), разные ERP-интеграции (8+ в LP, 0 в Fortis), разные языковые и локализационные требования.

### Влияние

**На кого влияет:**

* **R&D** (206 FTE) — двойные затраты на разработку и поддержку идентичного функционала
* **Product** — два изолированных продуктовых процесса, каждый видит только свой рынок. Нет единого продуктового видения, нет переноса решений и лучших практик между рынками
* **Бизнес** — масштабирование на новые рынки заблокировано: LP привязан к РФ, Fortis захардкожен под Дубай — ни один из продуктов не готов к новому рынку
* **Клиенты** — разный UX, разный набор фич в зависимости от рынка

**Стоимость бездействия:**

* Дублирование R&D ресурсов не позволяет достичь целевой эффективности
* Каждый новый рынок = новый продукт с нуля — ни LP, ни Fortis не способны принять дополнительный регион без переработки
* AI & Data-first стратегия невозможна без единой data model
* Дублирование багов и security-уязвимостей в двух кодовых базах

**Блокируемые инициативы:**

* Консолидация R&D для повышения эффективности
* Выход на новые рынки (KZ, UZ и др.)
* Единая AI/Data платформа
* Public API для 3rd-party developers

**Масштаб:** затрагивает все R&D-стримы (MP, POS, Payments, VAS/Store), Platform Engineering, DWH/Data, QA, Support.

Решение
-------

Предлагается **расширить Fortis** до единой модульной платформы Global Merchant Portal. Не мержим два продукта, не создаём новый с нуля — берём Fortis как основу(baseline) и переносим LP-специфику (compliance, ERP-интеграции, granular permissions) как подключаемые модули. Fortis выбран как основа потому, что архитектурно чище: Keycloak (industry standard IdP), BFF pattern, cursor pagination, structured data model, Fee system. Экономия ~6 месяцев по сравнению с greenfield.

Архитектура строится на трёх слоях: **Core** (единый для всех рынков, reuse ≥ 75%), **Market** (локальные адаптеры через provider pattern без правок Core) и **Store** (VAS через Open API без правок Core). Ключевой принцип — **Modules, not Regions**: нигде в системе нет понятия «регион», есть модули, которые включаются per company (`company.modules.X`). Новый рынок = новый market preset (JSON-конфиг) + при необходимости новые compliance providers. Код Core не трогаем. TTM North Star: ~12 недель на новый рынок.

MP реализуется как **shell-приложение** (layout, routing, auth, shared services), а POS, Payments и VAS/Store поставляют свои lazy-loaded модули в shell. 4 stream-aligned команды работают параллельно: Global MP (shell + core), Global POS (sales, terminals, ref), Global Payments (provider abstraction, ref), Global VAS/Store (loyalty, menu, shop, surveys).

### Образ результата (Core GA)

**Proof point:** новый мерчант из РФ регистрируется на Global MP, создаёт компанию, настраивает юрлицо, работает с каталогом, заказами и складом — на единой кодовой базе с MENA (разные деплойменты, данные в своих юрисдикциях).

**Платформа:**

* Два региональных деплоймента (MENA + RU) на одной кодовой базе
* Новый рынок = market preset (локаль, валюта, набор модулей) + при необходимости compliance-адаптеры, 0 правок Core. TTM North Star: ~12 недель
* Reuse Rate ≥ 75% Core-сервисов

**Явно вне scope:** миграция существующих LP-мерчантов — отдельный проект. LP продолжает работать в режиме feature freeze.

Альтернативы
------------

| Вариант | Плюсы | Минусы | Причина отклонения |
| --- | --- | --- | --- |
| **A. Greenfield** — новый продукт с нуля | Чистая архитектура, нет legacy, полная свобода выбора стека | 18+ мес, 3 продукта одновременно (LP + Fortis + новый), бюджет не позволяет, тройное дублирование на переходный период | Срок и бюджет несовместимы с целевым планом трансформации. Fortis покрывает ~70% целевой архитектуры из коробки |
| **B. Fork LP** — взять LP как основу | LP имеет compliance (фискализация, маркировка, ЕГАИС), 8+ ERP-интеграций | Tech debt: Custom JWT (нет SSO, MFA), page-based pagination, PascalCase, вложенные объекты, ~12 мес | Fortis data model чище (Keycloak, cursor, BFF, structured addresses). LP auth не масштабируется на новые рынки |
| **C. Fortis + extend** (выбран) | Keycloak (industry standard), BFF, cursor, Fee system. Экономия ~6 мес vs greenfield | LP compliance переносится как модули (Market-проекты ≤ 3 мес дополнительно) | **Принят** |
| **D. Adapter** — два бэкенда за фасадом | Быстрый старт (~9 мес), минимальные изменения в обоих продуктах | Два бэкенда живут вечно, latency трансформации, двойная поддержка навсегда, не устраняет корневую проблему | Не устраняет дублирование и не повышает эффективность R&D. Два бэкенда остаются навсегда |

Техдолг
-------

### Осознанный техдолг

* **LP в read-only** — два деплоймента одновременно (LP + Global). LP продолжает работать для существующих клиентов. Decommission: после миграции LP-мерчантов (отдельный проект)
* **NFR не валидированы** — данных с production нет, нагрузки и инфра-стоимость определяются на Discovery. Риск пересмотра инфра-решений
* **DWH-архитектура не определена** — РФ-отчёты работают на OLAP с большими объёмами. Решение — в Discovery

### Компромиссы

| Компромисс | Причина | Митигация |
| --- | --- | --- |
| Core GA без фискализации и маркировки | Risk isolation: compliance-специфика не блокирует запуск Core | Market-проекты (≤ 3 мес каждый) стартуют параллельно с финальной фазой Core |
| LP продолжает работать параллельно | Миграция существующих LP-мерчантов — сложный проект, не должен блокировать Core | LP в режиме feature freeze с Day 1, только critical/security fixes. Dedicated LP support |
| Shell architecture: overhead координации 4 стримов | Альтернатива (монолит/одна команда) = bottleneck и конфликты приоритетов | Shell contract freeze в начале проекта, затем стримы автономны. Architecture Council weekly |
| Keycloak для LP-пользователей (новый auth flow) | Custom JWT не масштабируется (нет SSO, refresh, MFA, social login) | Custom SPI для phone+OTT, переходный период с dual-token. UAT с реальными пользователями |
| Cursor pagination (LP-клиенты на page-based) | Cursor стабильнее и быстрее для больших данных | BFF обеспечивает трансляцию page → cursor на переходный период |

Влияние на команды
------------------

### Прямые участники

| Команда / Роль | Влияние | Действия |
| --- | --- | --- |
| **LP** | Высокое | Full-stack команда. Переход на Fortis-кодовую базу: cross-training Keycloak, BFF pattern, новые API-конвенции. Keycloak JWT validation, API consumer audit — prerequisite. Часть команды — на LP support |
| **Self-Service** | Среднее | Расширение MP shell для multi-market. Новые модули (inventory, legal entity). Module integration points для POS, Payments, VAS |
| **Core Services** | Среднее | API v1 extension, Keycloak Custom SPI (phone+OTT), compliance stubs, provider pattern implementations, BFF Config Profile |
| **POS Team** | Высокое | Sales module для MP shell. RU Compliance (fiscal, marking) |
| **Payments Team** | Среднее | Provider abstraction. Integration module для MP shell |
| **CRM&Loyalty** | Среднее | Модуляризация Fortis VAS. Перенос функциональности Surveys, CRM из LP |

### Затронутые смежные функции

| Функция | Влияние | Действия |
| --- | --- | --- |
| **DevOps / Platform** | Высокое | Multi-region deploy (MENA + RU). Keycloak-инфраструктура. CI/CD для единого pipeline |
| **DWH / Data** | Высокое | ETL pipeline из Global OLTP в DWH. Решение по OLAP для РФ-отчётов. Архитектура — в Discovery |
| **Support** | Среднее | Новые сценарии: единый auth, два деплоймента, переходный период LP |
| **Admin / Back-office** | Среднее | Ролевая политика, тарифы, подписки — через единый permission model |
| **QA** | Высокое | Multi-config testing (presets x modules x roles). Cross-stream E2E |
| **Security** | Среднее | Keycloak review. Public API security (OWASP API Top 10). Pen testing |
