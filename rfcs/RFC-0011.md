# RFC-0011: Интеграция Pyrus и 1С через n8n

> **Источник:** [Confluence](https://bz.incdb.team/pages/viewpage.action?pageId=3421831337)

**Автор:**

**Дата создания:**

**Статус:** APPROVED

---

Проблема
--------

В текущем процессе согласование заявок происходит в Pyrus, а бухгалтерский учёт и проведение платежей — в 1С, при этом обмен данными между системами частично выполняется вручную. Это приводит к дублированию данных, ошибкам и отсутствию сквозной связки между заявкой, платежом и бухгалтерской проводкой.

---

Предлагаемое решение (1 параграф + схемы)
-----------------------------------------

### Общая архитектура

Предлагается реализовать интеграцию через выделенный интеграционный узел (VM `pyrus-raum.life-pay.tech` в RBS Cloud), который выступает центральным оркестратором взаимодействия между системами Pyrus, 1С, клиент-банком и внешними сервисами.

Архитектура состоит из двух основных компонентов:

* **Интеграционный узел (VM pyrus-raum.life-pay.tech)**
  + Pyrus Integration Service
  + n8n (workflow orchestrator)
* **Сервер 1С (VM 1C)**
  + бизнес-логика
  + бухгалтерский и управленческий учёт
  + справочники контрагентов и договоров

Интеграционный узел выполняет:

* приём входящих событий из Pyrus (webhook/API)
* оркестрацию интеграционных сценариев
* трансформацию и маршрутизацию данных
* взаимодействие с API 1С и внешними сервисами

n8n используется как основной механизм реализации бизнес-процессов и интеграционных workflow.

---

### Data Flow

Интеграция построена по событийной модели с двусторонним обменом данными:

1. **Pyrus -> Интеграционный узел**
   * протокол: HTTPS (TCP 443)
   * назначение: webhook-события, API вызовы
2. **Интеграционный узел -> 1С**
   * протокол: SSL / HTTPS
   * назначение: передача данных о договорах, синхронизация справочников, передача платёжных данных
3. **1С -> Интеграционный узел**
   * протокол: SSL / HTTPS
   * назначение: обратная синхронизация, обновление статусов
4. **Интеграционный узел -> внешние сервисы**
   * протокол: HTTPS / API
   * назначение: интеграция с клиент-банком, взаимодействие со сторонними сервисами

---

### Бизнес-процесс

Реализуется сквозной процесс:

1. Заявка создаётся и согласуется в Pyrus
2. Событие поступает в интеграционный узел
3. В n8n запускается workflow
4. Формируется платёжное поручение
5. Данные передаются в 1С
6. Операция отражается в учёте
7. (опционально) статус возвращается обратно

Ключевой инвариант:

> Проведение платежа возможно только при наличии согласованной заявки в Pyrus

---

### Синхронизация данных

* Контрагенты могут создаваться в обеих системах
* 1С является источником уникального идентификатора
* Идентификатор сохраняется в Pyrus
* Выполняется регулярная двусторонняя синхронизация

Передача договоров включает: реквизиты, контрагента, суммы и сроки, финансовые аналитики.

---

### Безопасность и ограничения

#### Сетевой уровень

* Все соединения — только по TLS (не ниже 1.2)
* Ограниченный набор cipher suites
* Whitelisting строго по конкретным IP (без диапазонов)
* Входящие соединения: HTTPS (TCP 443) — интеграция, SSH (TCP 22) — администрирование

#### Доступ к серверу

На интеграционном узле:

* запрещена password authentication
* используется только аутентификация по ключам
* отключён root login
* ограничен список пользователей (AllowUsers)
* ведётся логирование ssh и sudo

#### Обмен данными

* интеграция с 1С — только по SSL/HTTPS
* двусторонний защищённый обмен

#### Особенности

* n8n как оркестратор может временно хранить данные выполнения workflow
* требуется контроль: доступа к системе, хранения данных, логирования операций

### Итог

Архитектура централизует интеграционную логику в одном узле, обеспечивает автоматизацию финансовых процессов и синхронизацию данных между системами, при этом 1С остаётся изолированной системой учёта с контролируемым доступом.

---

Альтернативы (кратко)
---------------------

* **Прямая интеграция (без n8n):** увеличивает сложность разработки и поддержки, снижает гибкость при изменении бизнес-процессов, привносит риски безопасности при прямой экспозиции чувствительных ресурсов 1С напрямую в интернет.
* **Вариант "ничего не делать":** сохраняются ручные операции, высокий риск ошибок и отсутствие сквозного контроля процессов

---

Компромиссы / Технический долг
------------------------------

* Использование n8n как центрального оркестратора создаёт зависимость от одного компонента. Рассмотреть использование HA схемы с несколькими оркестраторами.
* Разработать DRP.
* Возможна временная запись чувствительных данных в workflow n8n
* Не гарантируется exactly-once доставка сообщений
* При росте числа сценариев увеличивается сложность сопровождения

Планируется проработка:

* политики хранения данных в n8n
* использование схемы оркестратора по отказоустойчивой схеме с дублированием n+1

---

Влияние на другие команды
-------------------------

* **Команда 1С:** потребуется поддержка API и обработчиков интеграции
* **Интеграционная команда:** разработка и поддержка workflow в n8n
* **DevOps/Infra:** поддержка в развертывании ресурсов
* **Финансы / бухгалтерия:** участие в тестировании и валидации бизнес-процессов
