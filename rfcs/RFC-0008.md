# RFC-0008: Интеграция Pyrus с 1С и HR-link для кадровых документов

**Проблема**
------------

Автоматизация кадровых документов (заявления на отпуск, командировку, увольнение, изменение персональных данных). Сейчас данные (ФИО, юрлицо, рабочая почта, табельный номер, остаток отпуска, вид занятости) передаются между 1С, Pyrus и HR-link вручную, что приводит к ошибкам и задержкам. Сотрудники заполняют заявления вручную в Hr-link, тратится **в 2 раза больше времени** на запуск и согласование процессов.

**Предлагаемое решение**
------------------------

Pyrus выступает входным окном для процесса: через оркестратор n8n запрашивает данные из 1С через REST API. Сотрудник формирует заявку в Pyrus, после согласования n8n отправляет данные в HR-Link для создания документа. HR-Link -> вебхук -> n8n -> обновление задачи в Pyrus (статус, PDF).
Корпоративный интеграционный шлюз на базе n8n, развёрнутый на существующей VM `pyrus-raum.life-pay.tech` (RBS Cloud). Pyrus — входное окно для сотрудника, n8n оркестрирует обмен, HR-link — система кадровых документов.

### Компоненты

| Компонент | Технология | Описание |
| --- | --- | --- |
| n8n | n8n (последняя стабильная) | Оркестратор: обработка вебхуков из Pyrus, вызовы API 1С и HR-Link, трансформация данных |
| Pyrus integration service | Самописный (Python/Node.js) или штатный вебхук-обработчик | Приём событий из Pyrus, передача в n8n |
| REST API 1С | 1С:ЗУП (HTTP-сервис) | Отдаёт данные сотрудников по запросу |
| REST API HR-Link | HR-Link (штатный API) | Создание документов, отправка вебхуков |

### Таблица потоков данных

| Действие | Направление потока | Описание потока |
| --- | --- | --- |
| **2. Получить данные сотрудника** | `Интеграционный шлюз (n8n) -> 1С:ЗУП (REST API)` | n8n синхронно (sync) запрашивает через HTTPS/REST API у 1С:ЗУП данные сотрудника (ФИО, юрлицо, табельный номер, остаток отпуска, вид занятости). API возвращает JSON-объект и записывает в справочник Pyrus |
| **3. Событие: заявка согласована** | `Pyrus -> Интеграционный шлюз (n8n)` | Pyrus отправляет вебхук (async) в n8n о том, что заявка сотрудника прошла согласование руководителя. |
| **6. Создание объекта в HR-link** | `Интеграционный шлюз (n8n) -> HR-link (REST API)` | n8n отправляет синхронный POST-запрос в HR-link с маппингом полей из Pyrus и полученными из 1С данными. HR-link создаёт кадровый документ и возвращает `document_id` |
| **8. Изменить заявку (записать id объекта)** | `Интеграционный шлюз (n8n) -> Pyrus (через API)` | n8n синхронно обновляет заявку в Pyrus: записывает в неё `document_id`, полученный от HR-link, ссылку на документ и статус |
| **13. Событие: документ подписан** | `HR-link -> Интеграционный шлюз (n8n)` | После подписания документа сотрудником HR-link отправляет асинхронный вебхук в n8n, сигнализируя, что документ подписан |
| **14. Изменить заявку** | `Интеграционный шлюз (n8n) -> Pyrus (через API)` | n8n синхронно обновляет заявку в Pyrus: меняет статус на «Документ подписан» |

**Альтернативы (кратко)**
-------------------------

* Альтернатива 1 (оставить ручной ввод): Не выбрана из-за риска ошибок и низкой скорости.
* Иные варианты не рассматривались. Если оставить без изменения, то сохранятся текущие проблемы

**Компромиссы / Технический долг**
----------------------------------

Нет значимого техдолга. Интеграция выполняется стандартными REST API и будет разработана с текущим подрядчиком, который делает доработку по бюджету.

**Влияние на другие команды**
-----------------------------

* Команда 1С: Предоставить REST API (данные по сотрудникам) с аутентификацией.
* Команда HR-link (КДП): Предоставить REST API для приёма документов.
* ИБ и Compliance: Согласовать передачу персональных данных (ФЗ-152).
* Инфраструктура: Размещение n8n на существующей VM (pyrus-raum.life-pay.tech) по аналогии с другими интеграциями.
