1. Проблема
-----------

### Что сломано

CCO (Максим Ерофеев) и CEO LifePay (Ашот Григорян) не имеют оперативного доступа к аналитике продаж. Каждый запрос проходит через аналитика: формулировка -> SQL -> дашборд -> интерпретация. Цикл занимает от нескольких часов до нескольких дней.

Существующие дашборды (Яндекс DataLens) покрывают стандартные отчёты, но не отвечают на ad-hoc вопросы: «какая конверсия у менеджера X за последние 2 недели по воронке Y?», «покажи аномалии по недозвону за апрель», «сравни unit economics по 6 воронкам».

### Конкретно

* Аналитик (Влад Сахаров) тратит ~30% рабочего времени на подготовку ответов для CCO/CEO
* Решения принимаются на основании устаревших данных (дашборды обновляются с задержкой)
* Ad-hoc запросы требуют написания SQL — CCO/CEO не могут получить ответ самостоятельно
* Нет единой точки доступа к данным: amoCRM, Mango, биллинг, ФОТ разрознены

### Почему это важно

Пилот AI-трансформации LifePay (спонсоры: Ашот Григорян, Максим Ерофеев) зависит от демонстрации бизнес-ценности AI до 17 апреля. AI Dashboard — проект с наибольшим бизнес-эффектом прямо сейчас: даёт CCO/CEO инструмент принятия решений без зависимости от аналитика.

### Последствия бездействия

* CCO/CEO продолжают ждать ответов от аналитика
* Решения по воронкам принимаются на основании интуиции, а не данных
* Пилот AI-трансформации не демонстрирует ценность к дедлайну

### Какие метрики улучшатся

| Метрика | Сейчас | Целевое |
| --- | --- | --- |
| Время получения ответа на ad-hoc запрос | часы-дни (через аналитика) | секунды (чат) |
| Доля запросов, требующих аналитика | 100% | <20% (стандартные скилы) |
| Время Влада на подготовку отчётов | ~30% рабочего времени | ~5% (только нестандартные) |

2. Предлагаемое решение
-----------------------

### Концепция

Развернуть **Claude Code CLI** на выделенном сервере Яндекс Клауд с подключением к БД LifePay (read-only). Аналитические сценарии описываются как **skills** — Markdown-файлы с формулами, источниками данных и контекстом. Claude Code выполняет SQL-запросы к БД и возвращает ответ на естественном языке.

### Компоненты

| Компонент | Технология | Описание |
| --- | --- | --- |
| Сервер | Яндекс Клауд (ВМ) | Отдельный сервер, не на существующей инфраструктуре |
| LLM | Claude Code CLI (Anthropic) | Модель opus, --print mode |
| Auth | oauth2-proxy + Google SSO | @life-pay.ru домен + whitelist email (бэкенд, не JS-пароль) |
| БД | PostgreSQL (read-only user) | SELECT-only на схему `lp_analytics_ai` (masked views без ПДн), timeout 30s, обязательный WHERE, LIMIT в SKILL |
| ФОТ | ClickHouse (read-only) | `1c_fot_transactions` через Илью Гринева |
| Skills | Markdown файлы (SKILLS.md) | Описание метрик: поля, формулы, источники, контекст. Лежат на сервере + Confluence LifePay |
| Визуал | HTML dashboard | Веб-интерфейс для CCO/CEO |
| Мониторинг | `/monitor` endpoint | health check: БД connection, uptime, last query timestamp |
| Логи | SQLite / файлы | Логирование всех запросов для анализа использования |

### Источники данных (read-only)

AI Dashboard работает не напрямую с исходными таблицами DWH, а через отдельную схему **`lp_analytics_ai`** с masked views, в которых удалены/захешированы ПДн (см. §5, ответ Алиеву Q3).

| Masked view (`lp_analytics_ai.*`) | Источник | Содержание | ПДн-обработка |
| --- | --- | --- | --- |
| `manager_kpi_masked` | `lp_analytics.ab_rozn_manager_kpi` | Лиды, SLA, конверсия | ФИО -> `manager_id` |
| `fin_dash_items_masked` | `lp_analytics.fin_dash_items` | Оплаченные счета, финансы, pen rate | Маскировка реквизитов контрагентов |
| `calls_masked` | `lp_analytics.calls` | Звонки Mango, агрегаты по длительности | Без номеров/текстов |
| `amo_calls_masked` | `amo_calls_latest_mv` | Счётчики звонков по воронкам | Без номеров |
| `amo_messages_masked` | `amo_messages_latest_mv` | Счётчики сообщений | Без текстов, без имён |
| `amo_leads_masked` | `amo_leads_latest_mv` | Лиды по воронкам, суммы, даты (точны с 01.09.2025) | Без ФИО/контактов |
| `amo_events_masked` | `amo_events_latest_mv` | События amoCRM, агрегаты | Без ФИО |
| `amo_companies_masked` | `amo_companies_latest_mv` | Счётчики/агрегаты по компаниям | Без названий/ИНН/контактов |
| `1c_fot_transactions` (ClickHouse) | 1С (ФОТ) | ФОТ-агрегаты по подразделениям | Агрегаты, без индивидуальных зарплат. Через Илью Гринева |

**Реализация masked-схемы — ответственность Алексея Ляпина (DevOps).** Требование добавлено в §6 как новый пункт v2.

### Skills (аналитические сценарии)

Влад Сахаров пишет skills по плану:

1. **БД** — для чего нужны, какие данные брать
2. **Поля** — какое что значит и как участвует в расчёте метрик
3. **Логика** — как считаются те или иные метрики
4. Оформляем в skill -> тестируем -> итерируем

Первый набор skills:

* `/salesdash` — общий дашборд (ab\_rozn\_manager\_kpi + amo\_leads)
* `/funnel-compare` — сравнение 6 воронок: конверсия, выручка, средний чек, скорость
* `/manager-rank` — ранжирование менеджеров по конверсии, выручке, CIR
* `/manager-deep` — глубокий разбор менеджера (ab\_rozn\_manager\_kpi + amo\_messages + amo\_events)
* `/anomalies` — мёртвые сделки, застрявшие воронки, падение конверсии
* `/weekly-report` — еженедельный отчёт с трендами и дельтами
* `/funnel-strategy` — стратегия по воронкам (ab\_rozn\_manager\_kpi + fin\_dash\_items)
* `/sales-portrait` — ФОТ по воронкам, CIR, стоимость продажи, KPI
* `/finance` — P&L, Cash Flow, ССЧ, ФОТ бюджет факт vs план

3. Почему не альтернативы
-------------------------

| Альтернатива | Почему нет |
| --- | --- |
| Расширение DataLens | Покрывает стандартные отчёты, но не ad-hoc запросы. Добавление нового отчёта = задача аналитику |
| Нейроаналитик DataLens | Рассмотрен (вопрос Аборина на Архкоме). Не подходит — нет прямого SQL-доступа к `lp_analytics` |
| Metabase / Superset | Self-service BI, но CCO/CEO всё равно должны знать SQL или строить визуализации. Порог входа высокий |
| Ничего не делать | CCO/CEO продолжают ждать аналитика. Пилот AI-трансформации не показывает ценность |

4. Что теряем (компромиссы и техдолг)
-------------------------------------

| Компромисс | Последствие | Мера управления |
| --- | --- | --- |
| Claude Code — внешний LLM (Anthropic) | Данные передаются в API Anthropic для генерации SQL | Read-only доступ. Без ПДн клиентов (ФИО/ИНН/телефоны). Только агрегированные метрики продаж (числа) |
| Skills пишутся вручную | Скорость зависит от Влада (~3 дня на первый набор) | Итеративный подход: начинаем с 9 базовых skills, дописываем по реальным запросам CCO |
| 1С ФОТ недоступен | Нет данных по зарплатам сотрудников в разрезе подразделений | Агрегированные данные через Илью Гринева (ClickHouse). Детализация — через отдельный запрос |
| Нет интеграции с OpenClaw | OpenClaw agents блокируются Anthropic при текущей схеме | Standalone Claude Code на отдельном сервере |
| Single instance, без резервирования | Возможен даунтайм при падении ВМ | OK — аналитический инструмент, не SLA-critical. `/monitor` endpoint для быстрого обнаружения |

### Технический долг

| Долг | Приоритет | Мера |
| --- | --- | --- |
| Skills в Markdown, не в БД | Низкий | Достаточно для 2 пользователей. При масштабировании — перенести в PostgreSQL |
| Нет автообновления данных | Средний | amoCRM views обновляются автоматически. Для 1С — ручная выгрузка |
| Логи в файлах, не в ELK | Низкий | Достаточно для пилота. При расширении — подключить к Data Platform |

### Руслан Алиев (10 апр, r.aliyev@life-pay.ru)

> **Q1:** «Какие именно данные клиентов могут уйти на серверы Anthropic и как это соотносится с требованиями по защите персональных данных?»
>
> **Ответ:** В Anthropic уходят **только агрегированные числовые метрики** (конверсия, выручка, счётчики по воронкам/менеджерам). ПДн клиентов (ФИО, ИНН, телефоны, email, адреса) **не передаются**. Запросы skills построены так, что SELECT возвращает только агрегаты (`COUNT`, `SUM`, `AVG`, `GROUP BY funnel/manager`). 152-ФЗ соблюдается: персональных данных субъектов ПДн в передаче нет.

> **Q2:** «Как защищён веб-дашборд от несанкционированного доступа?»
>
> **Ответ:** Три уровня защиты: (1) Google SSO через oauth2-proxy + whitelist по email (@life-pay.ru), (2) TLS (Let's Encrypt), (3) аудит-лог всех запросов в SQLite с привязкой к пользователю SSO. Изначально доступ у 4 пользователей (Ерофеев, Григорян, Николаев, Бадягин) + support (Шмонов). Управление whitelist — через Admin-панель у ИБ.

> **Q3:** «Что мешает AI-модели сгенерировать запрос, который вернёт чувствительные данные или перегрузит базу?»
>
> **Ответ:** Защита в два слоя — **защита от ПДн** и **защита от нагрузки**.
>
> **Слой 1 — ПДн на стороне БД (данные физически не могут покинуть LifePay):**
>
> Часть исходных views содержит ПДн, которые **нельзя передавать в Anthropic**:
>
> * `amo_messages_latest_mv` — тексты сообщений клиентов
> * `amo_companies_latest_mv` — названия, ИНН, контакты компаний
> * `amo_leads_latest_mv` — ФИО сделок/контактов
> * `amo_calls_latest_mv` — телефоны
> * `lp_analytics.ab_rozn_manager_kpi` — ФИО менеджеров (работники LifePay = ПДн сотрудников)
>
> **Мера:** Алексей Ляпин (DevOps) создаёт **отдельную схему `lp_analytics_ai`** с masked views поверх исходных. В masked views:
>
> * Удалены или захешированы ПДн-колонки (тексты сообщений, ФИО, телефоны, ИНН, email)
> * ФИО менеджеров заменены на `manager_id` (числовой идентификатор) — маппинг id<->ФИО хранится в skill-контексте только для интерпретации ответа **на стороне сервера**, в Anthropic не уходит
> * Строковые поля, где может быть свободный текст от клиента, — исключены из GRANT
>
> SELECT-only юзер AI Dashboard имеет GRANT **только на `lp_analytics_ai.*`**, не на исходные `amo_*_latest_mv` и не на `lp_analytics.ab_rozn_manager_kpi` напрямую. Таким образом, даже при произвольном `SELECT *` из любой доступной таблицы ПДн в результат не попадают.
>
> **Слой 2 — нагрузка и защита от «плохого SQL»:**
>
> 1. `statement_timeout = 30s` на уровне роли PostgreSQL
> 2. `work_mem` / `max_parallel_workers_per_gather` — ограничены для этой роли
> 3. В каждом SKILL явно прописан `LIMIT` и обязательный `WHERE` (фильтр по дате/воронке/менеджеру)
> 4. Skills работают как параметризованные шаблоны — Claude Code не пишет произвольный SQL с нуля, а подставляет параметры в заранее написанные Владом Сахаровым SKILL-шаблоны
> 5. Skills ревьюятся Владом перед публикацией — бизнес-логика и SQL-шаблоны заранее проверены
> 6. Аудит-лог всех SQL-запросов с привязкой к SSO-юзеру (SQLite на сервере)
>
> **Итог:** row-level ПДн физически не могут попасть в запрос -> не могут уйти в Anthropic. Нагрузка на DWH ограничена timeout, LIMIT и ревью skills.

> **Q4:** «Какова стоимость решения в месяц и где физически обрабатываются данные?»
>
> **Ответ:**
>
> * **Anthropic API:** ~$100-200/мес при пилотной нагрузке (4 пользователя LifePay + support, ad-hoc запросы)
> * **Инфраструктура:** ВМ Яндекс Клауд ~$20-30/мес (2 vCPU / 4 GB / 20 GB SSD)
> * **Итого:** ~$120-230/мес
> * **Данные физически:** SQL-запросы выполняются на ВМ в Яндекс Клауд (Россия). Промпт + агрегаты (без ПДн) отправляются в Anthropic API (data centers USA). ПДн клиентов **не покидают инфраструктуру LifePay**.

> **Q1:** «Раскрой, о какой БД речь. Нагрузка на БД — что с ней делаем?»
>
> **Ответ:** Речь про DWH LifePay (согласовано с Алексеем Ляпиным). Подключение read-only к следующим таблицам/view:
>
> * 8 таблиц/view `lp_analytics.*`: `ab_rozn_manager_kpi`, `fin_dash_items`, `calls`, `amo_calls_latest_mv`, `amo_messages_latest_mv`, `amo_leads_latest_mv`, `amo_events_latest_mv`, `amo_companies_latest_mv`
> * **Добавлена в v2:** `1c_fot_transactions` (ClickHouse, через Илью Гринева) — ФОТ агрегаты
>
> **Управление нагрузкой:**
>
> * SELECT-only юзер с GRANT только на перечисленные объекты
> * `statement_timeout = 30s` на уровне роли
> * Каждый SKILL содержит явные `WHERE` (фильтр по дате/воронке/менеджеру) и `LIMIT`
> * Запросы агрегированные (`GROUP BY funnel/manager/date`) — не возвращают row-level данные

> **Q2:** «Также непонятен скоуп: всю БД будем сканить?»
>
> **Ответ:** **Нет, скан всей БД исключён.** Запросы формируются только через SKILL-переменные (предопределённые параметризованные шаблоны, написанные Владом Сахаровым). Claude Code не генерирует произвольный SQL с нуля — он подставляет параметры в существующие SKILL-шаблоны. При хорошо написанных skills нагрузка контролируется на этапе ревью шаблона, а не во время исполнения.

> **Q3:** «Где лежат skills?»
>
> **Ответ:** На том же сервере Яндекс Клауд (в Git-репозитории) + дублирование в базе знаний Confluence LifePay для бизнес-ревью.
