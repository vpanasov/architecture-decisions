1. Проблема
-----------

### Что сломано

Менеджеры продаж LifePay тратят **~330 часов в месяц (2+ FTE)** на ручной набор номеров и ожидание гудков. 22,5% исходящих звонков заканчиваются пустыми гудками — 87 000 неуспешных наборов за 8 месяцев. Разброс fail rate между менеджерами — 3,4x (от 10,8% у Хафизова до 37,2% у Казаковой), что указывает на проблему процесса, а не навыков продаж.

### Конкретно (данные: 756 528 звонков Mango Office, авг 2025 — мар 2026)

* **22,5% исходящих** — пустые гудки. Менеджеры слушают гудки ~50 ч/мес
* **33,3% исходящих** — длительность <5 сек (сбросы, автоответчики)
* **20,2% сделок** — только 1-2 попытки дозвона при норме 3-8
* **531 сделка застряла в статусе «Недозвон»**, 399 из них — в воронке UPSALE триалов (клиенты, которые уже пользуются продуктом)
* **Ручной набор:** ~25 секунд на звонок vs ~4 секунды при автодозвоне = потеря 281 ч/мес
* **Разброс fail rate между менеджерами:** 3,4x — единый алгоритм устранит дисперсию

### Почему это важно

* Потенциал: **+40-70 дополнительных продаж/мес** (+320-560K руб/мес, или +3,8-6,7M руб/год)
* ROI: **4-10x**, окупаемость 1-2 месяца
* Пилот AI-трансформации LifePay: автодозвон — один из 4 ключевых проектов, ожидаемый CCO Максимом Ерофеевым к 17 апреля

### Последствия бездействия

* 2+ FTE продолжают сжигаться на пустые гудки
* Менеджеры с высоким fail rate остаются без коррекции процесса
* 531 сделка в «Недозвоне» не обрабатывается по SLA воронки

### Какие метрики улучшатся

| Метрика | Сейчас | Целевое |
| --- | --- | --- |
| Пустые гудки | 22,5% | ~0% (менеджер разговаривает только с ответившими) |
| Время на один набор | 25 сек | 4 сек |
| Разброс fail rate | 3,4x (10,8%-37,2%) | Единый алгоритм, целевой fail rate 12-15% |
| Сделок с 1-2 попытками | 20,2% | Автоматический SLA-перезвон |
| Доп. продаж/мес | 0 | +40-70 |

2. Предлагаемое решение
-----------------------

### Концепция

MVP автодозвона: **виджет amoCRM** (JS, встроен в карточку сделки) собирает лиды и задачи из amoCRM через браузерную сессию менеджера и отправляет JSON-запросом на **собственный backend на Яндекс Клауд**. Backend обогащает данными из Spark Interfax / DaData, ранжирует **собственной ML-моделью**, строит приоритизированную очередь и отправляет пачку номеров на параллельный прозвон в **Mango Office**. На продакшн-сервере **не нужны OAuth-токены amoCRM** — все данные из CRM приходят от виджета. Голосовой робот (встроенный в Mango Office) спрашивает клиента о готовности, при ответе «Да» соединяет с менеджером.

Все ML-скоринг и приоритизация выполняются локально на сервере. **Внешние LLM-сервисы (Anthropic и др.) не используются.**

### Что нового в решении (ответ на вопрос Гора Авакяна)

| Новое | Статус |
| --- | --- |
| **Собственный backend** (Python/FastAPI) на отдельной ВМ в Яндекс Клауд, папка DMZ. Принимает данные от виджета (не ходит в amoCRM напрямую). | Код готов (27 тестов), ждёт сервер |
| **ML-модель приоритизации лидов** (scikit-learn, GBM + логрег, 11 моделей по воронкам) | Обучена, 79,5% accuracy |
| **Виджет для amoCRM** (frontend) — кнопка «Готов к обзвону» + сбор лидов/задач из amoCRM через браузерную сессию менеджера | Готов (UI + сбор данных + POST на backend) |
| Интеграция **Mango Office API** (параллельный прозвон, голосовой робот) | Настройка на стороне Mango — Эмин Бадягин |

| Существующее, переиспользуется без изменений | Статус |
| --- | --- |
| amoCRM (источник лидов и задач) | Используется в проде LifePay |
| Sensei Workplace (формирует задачи менеджерам) | Уже интегрирован с amoCRM |
| Mango Office (текущий поставщик телефонии) | Уже используется в LifePay |

### Компоненты

| Компонент | Технология | Описание | Статус на 15 апр |
| --- | --- | --- | --- |
| amoCRM Widget | JavaScript (frontend) | Кнопка «Готов к обзвону» + сбор лидов/задач из amoCRM через браузерную сессию менеджера -> POST JSON на backend | Готов (UI + логика сбора + POST) |
| Backend API | **Python / FastAPI** | Принимает данные от виджета (`POST /api/widget/start`), обогащение -> ML -> очередь -> Mango. Не ходит в amoCRM напрямую. | Код готов (27 тестов), ждёт сервер |
| Secrets store | `.env` файл, root-only, chmod 600 | Токены Mango, Spark, DaData (OAuth amoCRM **не нужны** на проде — данные приходят от виджета) | Создаётся при деплое |
| Обогащение | Python + REST API Spark/DaData | ИНН -> ОКВЭД, возраст, ОПФ, проверка номеров | Готов (DaData real client) |
| ML-модель | scikit-learn (GBM + логрег) | 11 моделей по воронкам, обучены на 165 785 сделках x 152 полях | Обучена, 79,5% accuracy |
| Очередь прозвона | Python | Ранжирование: score x оптимальное время дня по часовому поясу | Готов |
| Mango Office API | REST, webhooks | Параллельный прозвон, голосовой робот, запись звонка | Настройка на стороне Mango (Эмин Бадягин) |
| Голосовой робот | Встроенный сценарий Mango | Простейший: «Готовы принять звонок от Life Pay? Да/Нет» | Настройка на стороне Mango |
| `/monitor` endpoint | FastAPI | Health check: uptime, БД, Mango API availability, last query | В разработке |
| Сервер | Яндекс Клауд ВМ, папка DMZ | Отдельная ВМ в DMZ (согласовано с Поляковым 14 апр) | Заявка #63035567 — согласована, ждём выдачу |

**Не используется:** никаких внешних LLM. Весь ИИ — локальная ML-модель (scikit-learn) на сервере в DMZ.

### ML-модель приоритизации

Отдельная модель на каждую из 11 воронок. Обучена на 165 785 сделках x 152 поля (июл 2025 — мар 2026). Используются только фичи, доступные в момент создания сделки (без leakage).

**Универсальные сигналы:**

| Фича | Вес | Описание |
| --- | --- | --- |
| `company_age_days` | 12-26% | Возраст компании по ЕГРЮЛ |
| `created_hour` | 5-27% | Час создания лида (09-17 = горячий) |
| `distance_from_moscow_km` | 5-18% | ЦФО конвертируется на 17% лучше |
| `is_returning` | ~15% | Повторный клиент |
| `opf_code` | до 14% | ООО конвертируется в 2-3x лучше, чем ИП |
| `okved_section` | 3-7% | Туризм, розница, еда — лучшие сегменты |

**Accuracy по воронкам (сводка):**

| Воронка | Accuracy | Spread TOP-10% vs BOT-10% |
| --- | --- | --- |
| ОРРК мониторинг | 0,979 | 17% |
| Розничные заявки | 0,904 | 88% |
| ОРДК: Проверка гипотез | 0,875 | 91% |
| Замена ФН | 0,841 | 71% |
| (А) UPSALE триалов | 0,818 | 75% |
| **Среднее по 11 воронкам** | **0,795** | **70%** |

**Spread 70%** означает: TOP-10% лидов по ML-скору конвертируются на 70 п.п. лучше BOTTOM-10%. Менеджер, получающий приоритизированную очередь, закрывает больше сделок за то же время.

### Интеграция с Sensei (ответ Алиеву Q5)

* Sensei Workplace уже используется в LifePay и формирует задачи менеджерам из лидов amoCRM (владеет lifecycle задач)
* Виджет amoCRM **собирает задачи из amoCRM** (сформированные Sensei) через браузерную сессию менеджера и передаёт на Backend. Backend добавляет ML-приоритизацию
* Backend **не создаёт задачи** напрямую — Sensei остаётся единственным owner задач
* Backend **не двигает этапы сделок** — это делает Sensei/AdminArea
* При ответе «Нет» или недозвоне — Sensei создаёт задачу на перезвон по SLA воронки через webhook от Mango

**Статус:** согласование проведено на звонке с Николаем Николаевым 14 апреля. Прямой вызов Sensei API из backend **не требуется** — виджет собирает задачи (созданные Sensei) из amoCRM через браузерную сессию.

### Интеграция с Mango Office

* Параллельный прозвон 3-5 номеров одновременно с множителем параллельности <=1,2 (консервативный, чтобы минимизировать abandon rate — см. ответ Алиеву Q2)
* Голосовой робот: сценарий «Готовы принять звонок от Life Pay? Да / Нет»
* При «Да» — поиск свободного менеджера -> входящий звонок через добавочный номер
* При «Да», но все менеджеры заняты — робот говорит «Мы перезвоним в течение N минут» -> создаётся задача в Sensei
* При «Нет» / недозвоне -> webhook -> Sensei ставит задачу на перезвон по SLA

**Статус:** настройка тарифа и подключение услуги — **Эмин Бадягин (LifePay)**, в работе в рамках существующего IT-бюджета LifePay. Отдельного согласования бюджета не требуется.

### Пилот

* **Масштаб:** 2-3 менеджера одной воронки (ОРДК, согласовано с Никитой Сухоруковым)
* **Длительность:** 2 недели после запуска backend
* **Ручной fallback:** менеджеры пилота продолжают иметь доступ к ручному набору. Кнопка «Готов к обзвону» — опциональная. Пилот не блокирует основной процесс продаж
* **Остальная команда (42 менеджера):** работает в штатном режиме без изменений

3. Почему не альтернативы
-------------------------

| Альтернатива | Почему нет |
| --- | --- |
| Купить готовую интеграцию Mango + Sensei + Мэдисон | 500K-1,5M руб, 3-4 месяца, 3 вендора. Нет ML-приоритизации. Можно сделать **после** успешного MVP как полноценное решение |
| Включить встроенный предиктивный набор Mango «из коробки» (Вариант 1 из презентации Николаева) | Засоряет очередь AdminArea, ломает существующую аналитику, нет контроля над приоритизацией. Николаев подтвердил риски |
| Использовать внешний SaaS-автодозвонщик (Скорозвон и аналоги) | Потребует миграции данных из amoCRM, нет интеграции с Sensei и ML-моделью LifePay |
| Нанять 2 дополнительных FTE | ~4M руб/год, не решает проблему пустых гудков и разброса fail rate |
| Ничего не делать | 330 ч/мес впустую, 87K пустых гудков, потеря 40-70 сделок/мес |

4. Что теряем (компромиссы и техдолг)
-------------------------------------

| Компромисс | Последствие | Мера управления |
| --- | --- | --- |
| MVP на 2 недели, не полноценная интеграция | Может упасть, не покрывает все edge cases | Feature flag, откат к ручному режиму за секунды. Основной процесс продаж не блокируется |
| ML-модель обучена на исторических данных | Accuracy может деградировать со временем | Мониторинг метрик, переобучение раз в квартал силами Codos AI |
| Зависимость от Mango API | Mango может изменить API или ограничить доступ | Абстракция `IDialer` в backend — замена провайдера телефонии без переписывания пайплайна |
| Голосовой робот — простейший сценарий «Да/Нет» | Не квалифицирует лида до соединения с менеджером | Достаточно для MVP. Расширение сценария — следующий этап после успешного пилота |
| Параллельная работа с текущим ручным процессом | Менеджеры вне пилота продолжают звонить вручную | Пилот на 2-3 менеджерах одной воронки, затем раскатка по остальным |
| Параллельный прозвон может оставить клиента без соединения | См. ответ Алиеву Q2 | Множитель <=1,2, robot-first подтверждение, автосоздание задачи на перезвон |

### Технический долг

| Долг | Приоритет | Мера |
| --- | --- | --- |
| Нет мониторинга ML drift | Средний | Добавить после запуска MVP |
| Бэкап ML-моделей | Низкий | Хранение в Git LFS |
| Нет автоматических retry при падении Mango API | Средний | Добавить exponential backoff после первой итерации |
| Логи в файлах, не в ELK | Низкий | Достаточно для пилота. При расширении — подключение к Data Platform LifePay |
