1. Проблема
-----------

Сотрудники используют ChatGPT и Claude через личные аккаунты в рабочих задачах. Это создаёт три конкретных проблемы:

1. **Утечка персональных данных.** ФИО клиентов, номера карт, паспортные данные уходят во внешние API без контроля со стороны компании.
2. **Нет учёта расходов.** Компания не знает кто, сколько и на что тратит токены. Нет бюджетных лимитов на пользователя или отдел.
3. **Нет единой точки входа.** Каждый сотрудник самостоятельно получает ключи — операционный хаос при масштабировании.

**Целевая аудитория:** ~100-200 сотрудников Life-Pay
**Репозиторий:** https://gitlab.life-pay.ru/g.trushkov/llm-gateway
**Сервис (MVP):** https://llm-gateway.analytics.fortisds.tech/ *(требуется учётная запись)*
---

2. Предлагаемое решение
-----------------------

Корпоративный LLM Gateway — набор Docker-контейнеров на внешнем VPS (MVP) с планируемым переездом в k8s инфры Life-Pay на следующем этапе.

Характеристики VPS: 4 vCPU, 8 ГБ RAM, 80 ГБ SSD.

### Компоненты

| Компонент | Технология | Роль |
| --- | --- | --- |
| **Nginx** | nginx:1.27-alpine | Reverse proxy, TLS (Let's Encrypt), rate limiting, блокировка file upload |
| **Open WebUI** | v0.8.10 | Веб-интерфейс для сотрудников, история диалогов, управление пользователями |
| **PII Masker** | Python/FastAPI (собственный) | Маскирование ПД через NER (SpaCy) перед отправкой во внешние API |
| **LiteLLM** | pinned SHA | API-агрегатор для Claude и GPT, бюджеты по пользователям, логирование токенов |
| **PostgreSQL** | 16.8-alpine | История диалогов, метрики LiteLLM, логи запросов PII Masker |
| **Watchdog** | curlimages/curl | HTTP health check каждые 5 минут, алерты на email (g.trushkov@life-pay.ru) через ntfy.sh |
| **Certbot** | certbot/certbot | Автоматическое обновление TLS-сертификата |

### Доступные модели

| Модель | Провайдер | Назначение |
| --- | --- | --- |
| Claude Haiku — Быстрый | Anthropic | Простые задачи, быстрые ответы |
| Claude Sonnet — Умный *(по умолчанию)* | Anthropic | Баланс скорости и качества |
| Claude Opus — Максимум | Anthropic | Сложный анализ, длинные тексты |
| GPT-5 | OpenAI | Альтернативный взгляд |
| GPT-5 Mini — Быстрый | OpenAI | Быстрые задачи на базе OpenAI |

### Бюджеты и лимиты

* Общий бюджет: **$1 000/месяц** (покрывает 100-200 пользователей x $5)
* На пользователя: **$5/месяц** — ограничивает расход одного пользователя, корректируется по итогам пилота
* Rate limiting: 10 req/s API, 30 req/s UI, не более 20 одновременных соединений с IP
* Ожидаемый прирост БД за MVP (1-2 месяца): 100-300 МБ (меньше 1 ГБ)

3. Архитектура
--------------

*(Диаграммы доступны в Confluence)*

> **Ключевое свойство:** персональные данные никогда не покидают периметр шлюза в исходном виде.
> Во внешние API уходят только обезличенные токены вида `[PER_1]`, `[CARD_1]`.
> Оригинальный текст хранится в истории диалогов Open WebUI на VPS — это осознанный компромисс MVP (см. раздел 5).

4. Почему не альтернативы
-------------------------

| Вариант | Причина отказа |
| --- | --- |
| **Корпоративный аккаунт ChatGPT/Claude** | Не решает проблему ПД в промптах; нет централизованного учёта по пользователям; нет гибкости по моделям |
| **Локальная LLM (Ollama + Llama/Mistral)** | Качество значительно хуже Claude Sonnet/GPT-5; требует GPU-серверов (CAPEX); долгий time-to-value |
| **Azure OpenAI / Yandex GPT** | Долгие корпоративные согласования; нет Anthropic в Azure; Yandex GPT уступает по качеству |

5. Компромиссы и технический долг
---------------------------------

| # | Компромисс | Риск | Митигация / план |
| --- | --- | --- | --- |
| 1 | Внешний VPS | Данные обрабатываются за пределами инфры LP | TLS end-to-end; ПД маскируются до выхода в API; переезд в k8s LP — следующий этап |
| 2 | ПД в истории диалогов Open WebUI | Оригинальные промпты хранятся в БД на VPS | Доступ только через шлюз (443); план — TTL на историю или шифрование at rest |
| 3 | Единый PostgreSQL | Нет HA, один инстанс на всё | Ручной бэкап по расписанию; managed PostgreSQL при переезде в k8s |
| 4 | NOPASSWD docker для CI/CD | gitlab-runner с широкими правами на docker | Убирается при переезде в k8s (CI pipeline без sudo) |
| 5 | Email/пароль вместо SSO | Нет интеграции с корпоративным IdP | Ручное управление пользователями на MVP; продакшн — OIDC/SAML |
| 6 | Нет маскирования изображений | При вставке картинки ПД не маскируются | File upload заблокирован на уровне Nginx (`POST /api/v1/files -> 403`) |
| 7 | Мониторинг через ntfy.sh | Зависимость от внешнего сервиса алертов | MVP-компромисс; продакшн — Prometheus + Alertmanager внутри инфры |
| 8 | HTTP proxy (AWS) для выхода в API провайдеров | Промежуточный узел через облачный AWS компании | Прокси внутри инфры LP |

### Технический долг до продакшна

* Переезд в k8s Life-Pay (прямой доступ к API провайдеров в инфре LP уже есть)
* SSO/OIDC интеграция с корпоративным IdP
* Managed PostgreSQL (HA, бэкапы, point-in-time recovery)
* Prometheus + Grafana вместо watchdog
* TTL или шифрование at rest для истории диалогов
* Helm chart для k8s-деплоя
* Разграничение доступа к моделям по отделам (LiteLLM Teams)

6. Влияние на команды
---------------------

| Команда | Характер взаимодействия |
| --- | --- |
| **ИБ** | Уже вовлечена |
| **DevOps / Инфраструктура** | Переезд в k8s LP на следующем этапе |
| **IT / Helpdesk** | Управление учётными записями (MVP — вручную через Admin Panel) |
| **Команды-пользователи** | Уточняются; обучение минимальное — UX аналогичен ChatGPT |

Gateway не заменяет существующие инструменты принудительно — **добровольный пилот**.

---

7. Что не входит в MVP
----------------------

* Мобильное приложение
* Интеграция с внутренними системами LP (CRM, ERP) через function calling
* Fine-tuning или RAG на корпоративных данных
* Маскирование ПД в ответах моделей (только в запросах)
* Аудит-лог на уровне сообщений (логируется факт маскирования, не сами данные)

8. Целевая архитектура (продакшн)
---------------------------------

MVP на VPS — временное решение для запуска пилота. Целевое состояние — развёртывание в k8s инфры Life-Pay.

### Что меняется

| Компонент MVP (VPS) | Компонент продакшн (k8s LP) |
| --- | --- |
| Nginx (reverse proxy, TLS) | Ingress Controller (уже есть в кластере) |
| Certbot (Let's Encrypt) | cert-manager (уже есть в кластере) |
| Watchdog + ntfy.sh | Prometheus + Grafana-on-call (уже есть в компании) |
| Email/пароль | SSO через корпоративный IdP (OIDC/SAML) |
| Единый PostgreSQL на VPS | Managed PostgreSQL (HA, PITR) |
| docker compose + shell CI | Helm chart + GitLab CI в k8s |

Что остаётся без изменений: Open WebUI, PII Masker, LiteLLM — те же контейнеры, та же логика.

### Роадмэп

| Этап | Что делается | Зависимости |
| --- | --- | --- |
| **MVP (сейчас)** | Пилот на VPS, 100-200 пользователей, ручное управление учётками | — |
| **Этап 2** | Переезд в k8s LP: Helm chart, Ingress, cert-manager, managed PostgreSQL | DevOps: namespace + network policies |
| **Этап 3** | SSO/OIDC интеграция с корпоративным IdP | IT: регистрация приложения в IdP |
| **Этап 4** | Prometheus + Grafana-on-call вместо watchdog; разграничение доступа по отделам | DevOps: подключение к существующему стеку мониторинга |
| **Этап 5** | RAG на корпоративных данных, function calling с внутренними системами | Определяется по итогам пилота |
