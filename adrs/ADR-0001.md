# ADR-0001: Выбор подхода к созданию глобального ИТ-продукта

> **Источник:** [Confluence](https://bz.incdb.team/pages/viewpage.action?pageId=3412164649)

**Статус:** Draft.

**Дата принятия:** 2024-06-20.

**Решение приняли:** [заполняется вручную]

Контекст
--------

Компания поддерживает несколько продуктовых систем на двух рынках (РФ и ОАЭ). Ключевые домены дублируются: две кассовые платформы (Fortis POS и Life POS; Life POS состоит из двух кодовых баз — Backend API и модуль NEW), две системы управления терминальной инфраструктурой (TMS), две административные панели (LifePay и Fortis Admin Panel), два личных кабинета мерчанта.

Анализ доменных моделей (346+ сущностей, 21 область пересечений) показывает: пересечение POS-платформ составляет 9 из 21 доменной области (~43% по числу областей; 10-15% по переносимому as-is коду). LifePay является системой-хабом с пересечениями во всех остальных системах (120+ сущностей, 34 попарных пересечения).

Каждый новый рынок при текущей модели требует создания параллельного стека. Компания исследует три новых региона (Латам, СНГ, ЮВА). Без трансформации каждый из них потребует собственного стека, что сделает масштабирование экономически нецелесообразным. Инвесторы ожидают единый продуктовый портфель, развертываемый на новом рынке в предсказуемые сроки.

Текущие характеристики:

* Выход на новый рынок: 12-18+ месяцев, полный стек
* Численность: 155 FTE (РФ: 104, ОАЭ: 51)
* Доля переиспользуемого кода: 10-15% (переносимый as-is)
* IaC в зачатках; наблюдаемость ограничена Prometheus
* iPSP (онлайн-платежи) в незавершённой миграции с монолита на микросервисы

Решение
-------

Принят **гибридный подход**: платформенная архитектура с подключаемыми модулями [Платформа + модули] + конфигурационная локализация [Единое ядро с конфигурацией] + региональная изоляция инфраструктуры [Мультитенантность с изоляцией по региону].

Конкретная реализация:

1. **Новое платформенное ядро** — проектируется с нуля, извлекая общие доменные паттерны из всех 7 продуктов. Архитектурный стиль — доменно-ориентированные сервисы: 3-5 крупных сервисов по доменным границам (POS, Административный, Терминальный, Финансовый), каждый из которых является модульным монолитом внутри. Целевая доля ядра — 70-80%.
2. **Рыночные модули** — вся рыночная специфика (фискализация ФФД, ЕГАИС, маркировка, СБП для РФ; расширенная лояльность, Aani, nGenius для ОАЭ) оформляется как подключаемые модули через стабильные контракты (`IPaymentMethod`, `IRegulatoryCheck`, `IFiscalProvider`, `ITerminalDriver`, `IOrganizationEnrichment`, `ILoyaltyProvider`).
3. **Региональная изоляция** — каждый рынок получает собственное развёртывание с изолированным хранением данных. Инфраструктура описывается как код (Terraform).
4. **LIFE STORE** — магазин приложений с публичным API для сторонних и внутренних разработчиков, превращающий платформу в экосистему с дополнительным источником дохода (revenue sharing).
5. **Изолированные компоненты** — процессинг (PSP-LP, C++) и онлайн-платежи (iPSP, Python) сохраняют текущие стеки и интегрируются с ядром через API-контракты.

Целевые показатели:

* Время развёртывания: тип A (близкая регуляторика) — 12-16 недель; тип B (новая фискализация) — 4-6 мес.; тип C (уникальная регуляторика) — 6-9 мес.
* TCO: субленейный рост с числом рынков
* Численность: целевое сокращение на 30-35% после завершения трансформации (~100-110 FTE)
* Оценка трудоёмкости: 790-1250 чел-недель (12-20 месяцев)

Последствия
-----------

**Плюсы:**

* Изменение общего домена — одна кодовая база, автоматическое развёртывание на все рынки
* Рыночная специфика инкапсулирована в модулях; добавление нового рынка не затрагивает ядро
* Единая Data Platform для аналитики и ML
* LIFE STORE создаёт экосистему
* Стоимость поддержки растёт субленейно с числом рынков

**Минусы:**

* Параллельное существование старых и нового продуктов в течение 12-24 месяцев; двойная нагрузка на команду
* Межсервисное взаимодействие (API + Kafka) добавляет операционную сложность
* LifePay (PHP, система-хаб, 120+ сущностей) — постепенная миграция 12-18 месяцев
* Три стека в долгосрочной перспективе: C#/.NET (ядро), C++ (PSP-LP), Python (iPSP)
* Feature freeze для текущих продуктов на период извлечения каждого домена

Альтернативы
------------

### Архитектурные подходы к созданию глобального продукта

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
| --- | --- | --- | --- |
| Единое ядро с конфигурационной локализацией (только конфигурация, без модулей) | Низкая стоимость входа; единая кодовая база | Не справляется, когда рынок требует принципиально иного пользовательского пути или глубоких интеграций с госсистемами (ЕГАИС, фискализация ФФД) | Различия между рынками выходят за пределы параметрической конфигурации; элементы подхода *включены* в гибридное решение |
| Белая метка с локальным оператором (White-label) | Быстрый выход через партнёра; адаптация — ответственность партнёра | Потеря контроля над UX; фрагментация версий; невозможность LIFE STORE | Противоречит стратегии экосистемы и контроля над продуктом |
| Локальные форки (Fork + merge back) | Минимальные начальные затраты; полная свобода адаптации | Стоимость поддержки растёт экспоненциально; обратный перенос со временем невозможен | Именно эту модель компания де-факто имеет сейчас (Fortis как форк Life POS); решение направлено на выход из неё |
