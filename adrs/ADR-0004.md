# ADR-0004: Global Merchant Portal — объединение Fortis и LP на базе единой модульной архитектуры

> **Источник:** [Confluence](https://bz.incdb.team/pages/viewpage.action?pageId=3413606413)

**Статус:** **Accepted**.

**Автор:**  .

**Дата:**

Context
-------

Компания эксплуатирует два отдельных мерчант-портала с непересекающимися кодовыми базами:

* **Fortis** (MENA) — Keycloak (OAuth2/OIDC), BFF pattern, cursor pagination, structured data model, Fee system, VAS (Loyalty, Menu, Shop, Appointments, AI Assistant). Задумывался как глобальный, но захардкожен под один рынок (Дубай).
* **LP** (Россия) — Custom JWT (phone+OTT), API v6 (PascalCase, page-based pagination, guid), фискализация, маркировка, ЕГАИС, 8+ ERP-интеграций, опросы.

**Ключевые метрики текущего состояния:**

| Метрика | Значение |
| --- | --- |
| FTE | 206 (LP: 112, Fortis: 94) |
| Reuse Rate | 0% — каждая фича пишется дважды |
| Совместимость стеков | Техническая ~80% (Angular 20, Standalone). Расхождения: auth, VAS-набор, compliance, ERP, локализация |
| Новый рынок | Невозможен — оба продукта привязаны к одному региону |

**Блокируемые инициативы:**

* Консолидация R&D для повышения эффективности
* Выход на новые рынки (KZ, UZ и др.)
* Единая AI/Data платформа
* Public API для 3rd-party developers

Decision
--------

**Расширить Fortis** до единой модульной платформы Global Merchant Portal. Не мержим два продукта, не создаём новый с нуля — берём Fortis как основу и переносим LP-специфику (compliance, ERP-интеграции, granular permissions) как подключаемые модули.

Архитектура строится на трёх слоях: **Core** (единый для всех рынков, переиспользование >= 75%), **Market** (локальные адаптеры через provider pattern без правок Core) и **Store** (VAS-плагины через Open API без правок Core).

MP реализуется как shell-приложение (layout, routing, auth, shared services), а POS, Payments и VAS/Store поставляют свои lazy-loaded модули в shell. Stream-aligned команды работают параллельно.

### Подрешения

1. **Fortis = baseline** — Keycloak, BFF, cursor pagination, structured data model. Экономия ~6 месяцев по сравнению с greenfield.
2. **Modules, not Regions** — `company.modules.X` вместо `if (region === 'ru')`.
3. **Keycloak + Custom SPI** — единый IdP для всех рынков. SSO, refresh tokens, MFA из коробки.
4. **Shell + lazy modules** — стримы работают параллельно, 0 shell changes при добавлении модуля.
5. **Provider pattern** — новый рынок = новый provider (e.g. `IFiscalizationProvider`), 0 Core changes.
6. **API v1 unified BFF** — `id`, cursor pagination, snake\_case.
7. **Service state, no NgRx** — оба проекта уже на signals/rxjs.
8. **Sale vs Order — два домена** — POS-продажа (Sale) и ecom-заказ (Order) имеют разные lifecycle.
9. **Registration: FE -> BFF -> Keycloak** — контроль UI + бизнес-логики.
10. **Business Flow Engine** — модули влияют на бизнес-процесс.
11. **DWH для отчётов (OLAP)** — РФ-отчёты работают на OLAP с гигантскими объёмами.
12. **Auth Provider per Market** — один Keycloak realm, разные auth flows per market preset.
13. **Payment ПОСЛЕ Fiscal** — фискальный чек создаётся ДО приёма оплаты (требование ФЗ-54).
14. **Loyalty split: deduction/accrual** — списание бонусов ДО оплаты, начисление ПОСЛЕ выполнения заказа.
15. **Social Auth через Keycloak Identity Broker** — Google, Apple, Yandex ID как дополнительные способы входа.

Alternatives Considered
-----------------------

| Вариант | Плюсы | Минусы | Причина отклонения |
| --- | --- | --- | --- |
| **A. Greenfield** — новый продукт с нуля | Чистая архитектура, нет legacy | 18+ мес, 3 продукта одновременно, бюджет не позволяет | Срок и бюджет несовместимы. Fortis покрывает ~70% целевой архитектуры |
| **B. Fork LP** — взять LP как основу | LP имеет compliance, 8+ ERP-интеграций | Tech debt: Custom JWT, page-based pagination, PascalCase, ~12 мес | Fortis data model чище. LP auth не масштабируется |
| **C. Fortis + extend** | Keycloak, BFF, cursor, Fee system. Экономия ~6 мес vs greenfield | LP compliance переносится как модули (Market-проекты <= 3 мес дополнительно) | **Принят** |
| **D. Adapter** — два бэкенда за фасадом | Быстрый старт (~9 мес) | Два бэкенда живут вечно, не устраняет корневую проблему | Не устраняет дублирование |

Consequences
------------

### Positive

* **Переиспользование >= 75%** Core-сервисов
* **TTM ~12 недель** на новый рынок
* **Стримы параллельно** — shell architecture обеспечивает автономность команд
* **Единый IdP** — SSO, MFA, social login из коробки через Keycloak
* **Единая data model** — основа для AI/Data-first стратегии и Public API
* **Экономия ~6 месяцев** по сравнению с greenfield-подходом

### Negative / Risks

* **LP в read-only параллельно** — два деплоймента одновременно
* **NFR не валидированы** — данных с production нет
* **DWH-архитектура не определена**
* **Overhead координации стримов**
* **Keycloak для LP-пользователей** — новый auth flow

### Accepted Technical Debt

| Техдолг / Компромисс | Причина | Митигация |
| --- | --- | --- |
| Core GA без фискализации и маркировки | Risk isolation | Market-проекты (<= 3 мес каждый) стартуют параллельно |
| LP продолжает работать параллельно | Миграция LP-мерчантов — сложный проект | LP в режиме feature freeze с Day 1 |
| Shell architecture: overhead координации | Альтернатива = bottleneck | Shell contract freeze в начале проекта |
| Keycloak: новый auth flow для LP-пользователей | Custom JWT не масштабируется | Custom SPI для phone+OTT, dual-token |
| Cursor pagination (LP-клиенты на page-based) | Cursor стабильнее для больших данных | BFF обеспечивает трансляцию page -> cursor |
