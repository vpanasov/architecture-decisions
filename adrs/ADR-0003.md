# ADR-0003: Складской и логистический контур - вне AdminArea

> **Источник:** [Confluence](https://bz.incdb.team/pages/viewpage.action?pageId=3412164800)

**Статус:** Draft

**Дата принятия:** 2026-03-25

**Решение приняли:** [заполняется вручную]

Суть решения
------------

Складской учёт и логистику **не тащим в AdminArea**. Реализуем отдельно, AdminArea читает данные через API только после факта отгрузки.

Это не вопрос архитектурных предпочтений — это вопрос субъектов права. Пока оборудование на складе, оно принадлежит LifePay. AdminArea работает с данными мерчантов. Это разные юридические субъекты, и смешивать их в одной системе — значит создавать налоговые, гражданско-правовые и таможенные риски.

Контекст
--------

AdminArea 2.0 — это CRM для работы с мерчантами. В процессе проектирования возник вопрос: куда поместить складской и логистический функционал.

Данные по домену:

* 32 000-52 000 терминалов в год (РФ)
* 5 сервисных партнёров, 80 городов
* Учёт сейчас: 1С + Excel + ручной контроль, нет SN-трассируемости
* С 1 мая 2026 — обязательная маркировка радиоэлектроники (Честный знак, ПП РФ No 1954)
* Для Fortis/UAE склад в 2026 не приоритет
* Исторически `stock` submodule, Верста24 и интеграция с 1С живут в legacy AdminArea

Проблема
--------

Склад и AdminArea — это **два разных субъекта права в одном домене**.

|  | До отгрузки (LifePay) | После отгрузки (Мерчант) |
| --- | --- | --- |
| Владелец оборудования | LifePay — балансодержатель | Мерчант — покупатель |
| Что хранит система | остатки, движение, резервы, SN, МОЛ | оборудование клиента, TID, транзакции |
| Регуляторный режим | ФНС, ФТС, Честный знак | договорное право |
| Документ-граница | — | подписанный УПД |

Пока оборудование не отгружено — мерчант не субъект. После отгрузки — LifePay больше не субъект по этой единице. Они не пересекаются.

```
LifePay (Субъект А)                              Мерчант (Субъект Б)
─────────────────────────────────                ──────────────────────────
Склад -> Резерв -> Готово к отгрузке   ──УПД──>   В пути -> У клиента -> В работе
                                     граница
                                     права
```

AdminArea — это домен Субъекта Б. Склад — домен Субъекта А. Объединять их не упрощение, а архитектурная ошибка.

Решение
-------

**Складской контур (Стрим 1, Стрим 2) и логистика (Верста24) реализуются вне AdminArea.**

AdminArea подписывается на событие `equipment.shipped` и только после этого создаёт связь терминал -> мерчант в карточке клиента. До этого момента AdminArea про это оборудование ничего не знает.

```
Склад (баланс LifePay)    equipment.shipped + УПД    AdminArea (данные мерчанта)
────────────────────────  ──────────────────────>  ──────────────────────────
статус «Готово к отгрузке»   право перешло             TID появляется
                                                        в карточке мерчанта
```

Реализация:

1. **Стрим 1 — автоматизация текущего процесса (готовое оборудование)** — отдельный WMS-сервис. Зона ответственности: оборудование уже собрано и находится на складе LifePay или у сервисных партнёров. Функции: остатки, движение между складами, резервы под заявки, SN-tracking, инвентаризация, статусы у 5 СП в 80 городах.
2. **Стрим 2 — поставка и производство** — 1С КА / ERP. Зона ответственности: всё до появления готового терминала. Функции: ГТД, приёмка комплектующих, BOM, передача в производство, серийный учёт, выпуск готовой продукции, маркировка Честный знак, проводки НДС, ЭДО/УПД.
3. **Logistics-сервис** — обёртка над Верста24: посылки, доставка, статусы
4. **Warehouse API + Logistics API** — read-only интерфейс для AdminArea

Последствия
-----------

Плюсы:

* AdminArea не тащит бухгалтерский контур и РФ-регуляторику
* Архитектура остаётся применимой для Fortis/UAE без изменений
* Стримы масштабируются независимо
* Коллизия субъектов права устранена системно
* `FUNC.16` и `BG.14` (Верста24) уже помечены «Выносим: ДА» — решение согласовано с реестром

Минусы:

* Нужно спроектировать Warehouse API и контракт событий (минимум: `equipment.shipped`, `equipment.returned`)
* Владелец WMS-сервиса (Стрим 1) не определён
* Вопрос master-data owner (AdminArea vs 1С) открыт — блокирует дизайн интеграции
* До реализации WMS часть функций остаётся в legacy AdminArea

Альтернативы
------------

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
| --- | --- | --- | --- |
| **Склад как модуль внутри AdminArea 2.0** | одна система, меньше интеграций | смешение субъектов права; РФ-регуляторика загрязняет архитектуру UAE; юридические риски (ст. 120 НК, ст. 458 ГК, ст. 15.12 КоАП) | воспроизводит проблему, от которой уходим |
| **Стрим 1 в AdminArea, Стрим 2 в 1С** | меньше сервисов | Стрим 1 до отгрузки — это баланс LifePay, не данные мерчанта; граница должна быть системной, а не логической | полумера, коллизия не устраняется |
| **Всё в 1С** | зрелая платформа, нет дублирования | нет операционных данных в реальном времени; интерфейс 1С неудобен для работы с СП | 1С закрывает Стрим 2, но не Стрим 1 |

Операционные проблемы при смешении контуров
-------------------------------------------

| Проблема | Что происходит | Сценарий |
| --- | --- | --- |
| Бухгалтерия работает по данным системы, а не по факту | Система фиксирует отгрузку не в тот момент, когда она произошла | Конец квартала, пиковая отгрузка 400 терминалов банкам. 3,2 млн руб выручки уходят в другой квартал |
| Система показывает оборудование у мерчанта раньше, чем оно реально передано | Менеджер привязывает терминалы к карточке клиента «под резерв» | Мерчант банкротится — конкурсный управляющий включает терминалы в конкурсную массу |
| Склад и CRM видят разные статусы | Два контура в одной системе дают противоречивую картину | Партия 200 терминалов: производство выпустило, но DataMatrix не пришёл |
| Нельзя установить где потерялось оборудование | Система не ведёт движение после передачи сервисному партнёру | СП Новосибирск получил 30 терминалов, 4 потеряны, система не знает где |
| Операции с товаром фиксируются раньше, чем он юридически доступен | Менеджер резервирует товар, который ещё не прошёл таможню | Поставка 2 500 терминалов из Китая, таможня держит 5 дней |
