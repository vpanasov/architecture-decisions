# ADR-0002: Выбор подхода к проектированию ядра платформы

> **Источник:** [Confluence](https://bz.incdb.team/pages/viewpage.action?pageId=3412164651)

**Статус:** Draft

**Дата принятия:** 2026-03-24

**Решение приняли:** [заполняется вручную]

Контекст
--------

В ADR-0001 принято решение о создании платформенного ядра с подключаемыми рыночными модулями. Данный ADR фиксирует выбор архитектурного подхода к проектированию самого ядра: как нарезать 22+ доменных модуля на сервисы, как организовать их взаимодействие и границы.

Исходные данные:

* 346+ сущностей, 21 область пересечений между 7 продуктовыми системами
* Целевая доля ядра — 70-80% (ориентир инвесторов); расчётная — 57-62% по числу доменных областей, 50-60% по объёму бизнес-логики
* 155 FTE (РФ: 104, ОАЭ: 51), два офиса, гибридная модель
* Стек ядра: C#/.NET, Kafka, PostgreSQL, MongoDB, Keycloak
* IaC в зачатках; наблюдаемость ограничена Prometheus
* Fortis POS — антипаттерн «1 сущность = 1 микросервис» (30+ entity-сервисов)
* Life POS — монолит без точек расширения (две кодовые базы)
* LifePay — PHP-монолит, система-хаб (120+ сущностей, пересечения со всеми системами)
* Терминал — сквозная сущность, представленная в 5 из 7 систем

Решение
-------

Принят подход **доменно-ориентированных сервисов**: 3-5 крупных сервисов по доменным границам, каждый из которых является модульным монолитом внутри.

Принципы взаимодействия:

* Внутри сервиса: внутрипроцессные контракты (интерфейсы, медиатор). Kafka не используется внутри одного сервиса
* Между сервисами: API и асинхронные события (Kafka)
* Каждый сервис имеет собственную базу данных (database-per-service)

Последствия
-----------

**Плюсы:**

* Опирается на существующий анализ доменных моделей (346+ сущностей, 21 область, типы A/B/C)
* Баланс между автономией команд (независимый деплой между сервисами) и простотой (модульный монолит внутри)
* Соответствие организационной структуре: 12 целевых команд по 6-20 FTE
* Модули внутри сервиса общаются через вызовы методов — нет overhead сетевых вызовов для связанных доменов
* Каждый сервис деплоится независимо — изменение в TMS не требует передеплоя POS

**Минусы:**

* POS-сервис с 16 модулями — потенциальный bottleneck: 4 команды работают в одном деплойменте
* Межсервисное взаимодействие (API + Kafka) требует контрактного тестирования, Saga для распределённых транзакций, мониторинга вызовов
* Границы доменов могут быть определены неточно — открытый вопрос «Торговые точки: POS или Админ?»
* Сложнее, чем единый монолит: операционная нагрузка 3-5 деплойментов x N регионов

Альтернативы
------------

| Альтернатива | Плюсы | Минусы | Почему не выбрана |
| --- | --- | --- | --- |
| **Единый модульный монолит** (Shopify-модель) | Нет Saga, контрактного тестирования; проще отладка; на 10-15% дешевле | Координация релизов при 100+ FTE; нет независимого масштабирования доменов; единая точка отказа | Заслуживает проверки через spike в Фазе 0 |
| **Средняя гранулярность (10-15 сервисов)** | Точные границы автономии; независимый деплой каждого домена | 30-45 деплойментов x N регионов; значительно больше контрактных тестов | Преждевременна при текущем состоянии DevOps |
| **Event Sourcing (точечно, для терминалов и транзакций)** | Элегантное решение «сущность в 5 системах»; полная аудируемость | Порог входа 3-6 мес.; eventual consistency для POS-сценариев | Рекомендуется spike для терминального домена (Фаза 0b) |
| **SPLE (формальная модель признаков)** | Формализует конфигурацию рынка; автоматизирует сборку | Тяжёлый upfront; окупается при 10+ рынков | Целесообразна при выходе на 4-5-й рынок (Фазы 3-5) |
| **Data Mesh (data contracts между доменами)** | Формализует схему событий Kafka; ownership данных у доменной команды | Культурный сдвиг; overhead при 3-5 сервисах | Рекомендуются как дополнение (Фазы 2-4) |
| **Composable / PBC (UI-контракты для модулей)** | Полная инкапсуляция; ускорение фронтенд-разработки | Сложность micro-frontends; необходима единая дизайн-система | Целесообразен при конвергенции фронтенда (Фазы 4-6) |
| **BCM (Business Capability Mapping)** | Язык бизнеса для обоснования перед инвесторами; стабильные границы | Размытые границы способностей | Комплементарен DDD. Полезен для приоритизации доменов (Фаза 0b) |
| **Микроядерная архитектура (Microkernel)** | Единый механизм расширения; естественная основа для LIFE STORE | Общие домены невариабельны — нет смысла выносить в плагины | Для VAS-слоя (LIFE STORE): подход принят |
| **Cell-Based Architecture** | Максимальная изоляция; blast radius ограничен ячейкой | 12+ ячеек; IaC незрелый; кадровый дефицит DevOps/SRE | При 5+ рынках и зрелом IaC (2028+) |
| **User Journey Driven** | Оптимизация под сценарий; минимум межсервисных вызовов | Дублирование данных; нестабильные границы | Домены перпендикулярны путям |
